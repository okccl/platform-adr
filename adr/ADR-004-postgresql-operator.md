# ADR-004: PostgreSQL Operatorの選択

## Context

前職のオンプレKubernetesクラスタではPostgreSQLの管理に
Crunchy Postgres Operator（PGO）を使用していた。
運用を続ける中で以下の2つの課題が顕在化した。

**ライセンス問題:**
PGOはv5以降のライセンス方針変更により、商用環境での
無償利用がグレーゾーンになった。直ちに問題が発生したわけでは
ないが、長期的なリスクとして代替ツールの検討が始まった。

**安定性の問題:**
WAL（Write-Ahead Logging）関連の不整合が現場で発生しており、
運用上の課題になっていた。ただしWALの挙動については
当時PostgreSQL 13という古いバージョンを使用していたことも
影響している可能性があり、Operatorの問題なのかバージョンの
問題なのかは切り分けができていなかった。

これらの課題を受け、Tanzuからの基盤移行プロジェクトを契機に
PostgreSQL Operatorの再選定を検討した。自分が比較調査を担当し、
クラウド移行の可能性も踏まえてクラウド親和性を選定基準の
ひとつとした。この調査結果を踏まえ、ポートフォリオでは
CNPGを採用した。

---

## Options

### Option 1: Crunchy Postgres Operator（PGO）継続
- 現場での運用実績がある
- pgBackRestによるバックアップ機能が成熟している
- **問題点:** ライセンス問題が解消されておらず、
  商用環境での継続利用にリスクが残る

### Option 2: EDB Postgres Operator
- EnterpriseDB社が開発する商用サポート付きOperator
- PGOと操作感が近く、現場からの移行コストが低い
- 商用サポートが明確で、ライセンス問題の解消になる
- **問題点:** 商用製品であるため継続的なライセンスコストが発生する。
  またOSSコミュニティの活発さではCNPGに劣る

### Option 3: CloudNativePG（CNPG）
- PostgreSQLコアコントリビュータが開発するOSS Operator
- Kubernetes APIと自然に統合されたk8sネイティブな設計
- CNCF Sandboxプロジェクト（2022年採択）として
  コミュニティが活発
- バックアップはbarman経由でS3互換ストレージに対応しており、
  MinIOやAWS S3をそのまま利用できる
- クラウド環境（EKS・GKE等）との親和性が高い

---

## Decision

**CloudNativePG（CNPG）を採用する。**

---

## Reasons

1. **ライセンスが明確なOSSである**
   CNPGはApache 2.0ライセンスで商用利用に制限がない。
   Crunchyのライセンス問題を根本的に解消できる。
   EDBは商用サポートが明確な点は評価できるが、
   継続的なコストが発生するためポートフォリオでの
   採用には適さない。

2. **k8sネイティブな設計で運用が安定しやすい**
   CNPGはKubernetes Operatorとして設計されており、
   クラスタの状態管理・フェイルオーバー・バックアップが
   すべてKubernetes APIを通じて宣言的に管理できる。
   現場でのWAL不整合はPostgreSQL 13の古さに起因する
   可能性もあったが、よりk8sネイティブな設計のOperatorに
   移行することで運用上のリスクを低減できると判断した。

3. **クラウド展開との親和性が高い**
   現場での比較調査時点でクラウド移行の可能性があり、
   クラウド親和性を選定基準のひとつとした。
   CNPGはAWS・GCP・Azureでの利用実績が豊富で、
   ポートフォリオのPhase 11（EKS展開）でも
   同じOperatorをそのまま使用できる。

4. **バックアップ設計がMinIOと自然に統合できる**
   CNPGはS3互換ストレージへのバックアップに対応しており、
   Phase 8で導入したMinIOをバックアップ先として
   そのまま使用できる。`backup.enabled: true`だけで
   定期バックアップが有効になる抽象化を`common-db`で実現した。

---

## Consequences

**ポジティブ:**
- Apache 2.0ライセンスで商用利用に制限がなく、
  長期的な運用リスクがない
- Kubernetes APIとの統合により、クラスタ状態・
  フェイルオーバー・バックアップをすべて宣言的に管理できる
- Phase 11のEKS展開でも同じOperatorを使用でき、
  ローカルとクラウドで運用手順を統一できる

**ネガティブ・トレードオフ:**
- EDBと比べると商用サポートがないため、
  重大な問題が発生した場合はコミュニティへの
  依存度が高くなる
- 現場ではCrunchyからの移行が完了しておらず、
  CNPGの本番運用経験はポートフォリオ上のものに留まる。
  実際の移行では既存データの移行手順や
  アプリケーションの接続先変更など、
  追加の検討が必要になる
- barmanによるバックアップ設計はpgBackRestと異なるため、
  Crunchy運用経験者には学習コストが発生する
